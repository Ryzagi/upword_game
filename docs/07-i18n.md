# 07 — Internationalisation

Strategy for UI strings, word corpora, and the translation feature.

## Scope

Three distinct localisable surfaces, each handled differently:

1. **UI strings** — chrome, buttons, labels, error messages. Translated via
   `react-i18next` from JSON locale files.
2. **Word corpora** — gameplay content. Localised per-language as standalone
   JSON files (see `05-words-corpus.md`).
3. **In-game translation bar** — a feature, not localisation. Calls the
   backend translation proxy on demand.

## UI strings

### Library

`react-i18next` with `i18next-browser-languagedetector`. Reasons:

- Mature, well-documented, near-zero runtime overhead.
- Pluggable backends if we ever want server-loaded translations.
- ICU MessageFormat plugin available for plurals / gender if we need it later.

Install:

```
npm i i18next react-i18next i18next-browser-languagedetector
```

### File layout

```
frontend/src/locales/
├── en/
│   ├── common.json     all UI strings, namespaced by area
│   └── rules.json      the rules content (markdown)
└── ru/
    ├── common.json
    └── rules.json
```

`common.json` is loaded synchronously at app bootstrap. `rules.json` is
lazy-loaded only when the rules modal is opened.

### Key naming

Hierarchical, dot-separated, scoped by feature area:

```jsonc
{
  "menu": {
    "title": "Word Describer",
    "create_room": "Create new room",
    "join_room": "Join",
    "nickname_label": "Nickname",
    "language": "Language"
  },
  "lobby": {
    "players": "Players ({{count}})",
    "settings_title": "Settings",
    "team_join": "Join Team {{name}}",
    "randomize": "Randomize teams",
    "start": "Start game"
  },
  "errors": {
    "room_not_found": "That room doesn't exist.",
    "nickname_taken": "That nickname is taken in this room."
  }
}
```

Conventions:

- Always use ICU-style placeholders (`{{name}}`), never string concatenation
  in code.
- Use plural rules (`_one`, `_other`) when count varies — i18next supports
  this natively.
- Server emits error **codes**, never user-facing strings. The client looks
  the code up in `errors.<code>`.

### Detection and override

1. On first load, detect from `navigator.language`.
2. Fall back to `en` for any unknown language.
3. User override in Settings → Preferences persists as
   `localStorage.app.language` and takes precedence.

### Adding a language

1. Create `frontend/src/locales/<code>/common.json` by copying `en/common.json`.
2. Translate each value (keep keys identical).
3. Add the code to `frontend/src/i18n/index.ts`'s `supportedLngs` array.
4. Add a corresponding `data/words.<code>.json` (see corpus doc) if you want
   gameplay in that language.

CI step in Phase 6: a `keys-parity` test that fails if any locale file has
missing keys compared to `en/common.json`.

## Word corpora

Strict separation from UI strings. The corpus JSON is loaded by the backend
based on the **room's** language, set at room creation. UI language and room
language are independent — you can play a Russian-corpus game with English UI.

Game settings include:

- `ui_language` — set by each player individually (in Settings).
- `room_language` — set at room creation, immutable for the room's lifetime.
  Affects only the corpus.

If you want this exposed in the lobby UI, surface it under the Settings
panel: "Word language: EN / RU". Default = host's UI language at room
creation.

## Translation bar (gameplay feature)

This is **not** localisation — it's a tool players use to translate hints
inside the game.

- Source/target language pickers default to room language ↔ player's UI
  language but can be overridden.
- Backed by `POST /api/translate` which proxies the configured provider
  (MyMemory in v1).
- Cache key includes `(src, dst, normalised_text)`; cached for 1 hour.
- Future enhancement: cost N points per call, deducted from the team
  balance.

## Server-side strings

Backend should not assemble user-facing strings. Anything user-visible is:

1. Either echoed exactly from a user (nicknames, team names, guesses), or
2. An error `code` from the documented set.

The frontend localises every code via `errors.<code>` in the locale file. If
a code is missing in a locale, fall back to English; the i18next setup
handles this automatically.

## RTL languages

Out of scope for v1 (EN, RU are both LTR). When we add Arabic / Hebrew:

- Set `dir="rtl"` on `<html>` when the active language is RTL.
- Tailwind's logical properties (`ps-4` not `pl-4`) make this almost free if
  we use them from the start. Worth doing in Phase 2.

## Pluralisation examples

Russian needs three plural forms (`one`, `few`, `many`). i18next handles
this natively when keys are suffixed:

```jsonc
{
  "lobby": {
    "players_one": "{{count}} игрок",
    "players_few": "{{count}} игрока",
    "players_many": "{{count}} игроков"
  }
}
```

Use `t('lobby.players', { count: n })` and i18next picks the right form.

## Date / number formatting

Use `Intl.DateTimeFormat` and `Intl.NumberFormat` with the active locale.
Don't hand-format dates. We don't have many dates in this app, but the
round-summary timestamp and any future history view should use it.
